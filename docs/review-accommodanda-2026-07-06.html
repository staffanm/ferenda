Verdict. The architecture is holding: layer boundaries are clean (verified by checker and by hand in every slice), the mechanical layer is green, SQL is parameterized throughout, and the authenticated edit surface is genuinely well built. The problems cluster in two places instead: incremental correctness — seven independent ways the harvest/freshness machinery silently and permanently loses data (one per vertical, plus the render layer) — and the untouched 07-02 punch list, whose recurring patterns (load-bearing asserts on untrusted data, non-atomic writes, per-vertical harvest copies) kept re-accreting in exactly the new code this review examined.

Theme 1 — Silent, permanent document loss in incremental pipelines

The dominant result of this review. Every vertical grew its own watermark/stop logic (the lib/harvest.py promotion has been flagged since 07-01 and re-flagged 07-02 as HIGH #11), and each copy has a distinct hole. All of these share one failure shape: the run reports success, the watermark advances, and a document is never fetched again without a manual --full.

HigheurlexLate-indexed case law permanently invisible to incremental runs

eurlex/download.py:176,237 — enumerate_celex starts the CELEX year walk at max(first_year, since.year), but a CJEU judgment's CELEX year is the case year; its work date lands at decision time, years later. With a 2025 watermark, 62020CJ… slices are never queried again, so a 2020-case judgment decided in 2025 never enters the corpus. The code documents this exact lag at :698–702 for the resume value — but not for enumeration.

Fix: decouple enumeration start-year from the date floor for caselaw (walk from first_year or a bounded lag; the SPARQL wdate FILTER prunes cheaply).

HigheurlexIncremental floor gives zero tolerance for CELLAR's out-of-order indexing

eurlex/download.py:605–615 — floor = max(high, run − window) can only move up from the highest work date seen, yet RECENCY_WINDOW exists precisely because CELLAR indexes documents months after their work date. An act dated below high but indexed later is permanently skipped. This also defeats the deliberate no-notice retry design at :503–509 (double-confirmed by two independent reviewers): a doc that once yielded no content (e.g. a placeholder TIFF) is never re-enumerated once high passes its date.

Fix: let the active-sector floor reach min(high, run) − window; the on-disk skip makes the re-enumerated window cheap. Related MEDIUM: the FILTER(?d >= …) silently drops wdate-less documents every incremental run — use FILTER(!BOUND(?d) || …).

HighdvInterrupted or --limit run poisons the next run's lookahead stop

dv/download.py:172–201 + lib/util.py:334–343 — a crashed or truncated incremental run correctly withholds the watermark but leaves its ~N freshly written records on disk. The next run's stop rule ("20 consecutive already-downloaded ⇒ done") trusts disk state, stops above the un-fetched backlog, and then saves the watermark past it. Everything below the crash point is permanently skipped.

Fix: a dirty-marker set at the start of any mutating run, cleared only on clean completion; dirty at startup forces the backfill path. Related MEDIUM: the walk orders by decision date, not publication date — a referat published >14 days after decision sorts below downloaded records and is never harvested (the AD 2006–2017 gap in REWRITE §4 is this exact pattern); a periodic cron'd --full should be a documented part of the contract.

HighforarbeteListing walk stops on a type-filtered-empty page, truncating pm/ds harvests — including --full

forarbete/download.py:160–170 — iter_listing terminates when a page yields no items after type filtering. For the two types sharing regeringen.se category 1325, a page consisting entirely of the sibling type's documents (an all-Ds page is very plausible in the Ds-dominated older range) reads as "listing exhausted": the walk ends, the watermark is written, totals look plausible, and everything deeper in the listing is permanently skipped — for backfills too.

Fix: terminate on the raw item count (or on page > ceil(TotalCount/pagesize) — TotalCount is already fetched).

Mediumforeskrift · avgEmpty enumeration "succeeds" and clobbers a good watermark with null

foreskrift/harvest.py:577–578, avg/download.py:246–247 — selector rot or an API change yielding zero items produces zero Skips, so watermark.save(None) overwrites a previously earned date. The run reports clean while the corpus silently stops updating.

Fix: treat 0-items-enumerated as a Skip; make HarvestWatermark.save reject None (never regress a watermark).

MediumforeskriftOne transient resolve failure + the 20-hit lookahead = a permanently stranded document

foreskrift/harvest.py:565–578 — a failed per-doc resolve is logged as "retryable", but the watermark still advances; on the next newest-first run, the 20 documents downloaded above it trip the lookahead stop before the walk reaches it. One 500 from an agency page = a permanent hole unless an operator reads one log line. Sibling LOW: a non-PDF body (WAF page served 200) is dropped with no log and a record is still written, so the doc is permanently metadata-only with zero trace (harvest.py:267,305).

Fix: a retry sidecar consumed at run start (the sfs/versions.py skip-sidecar pattern), or withhold the watermark when errors > 0 — which is exactly what forarbete/riksdagen.py:336 already does; the two policies in one vertical answering the same question oppositely is itself flagged (LOW).

MediumremisserStub-record protection covers only HTTPError; a DOM-shaped failure permanently hides the case

remisser/download.py:310–321 — the stub design exists precisely to keep failures visible, but a 200-with-unexpected-DOM (bot interstitial) raises past the HTTPError catch after newer slugs were written; the next incremental run stops at the first on-disk slug and the failed case is never fetched again.

Fix: write the stub for any per-case fetch/parse failure (catch RequestException + a typed parse error), log, count as failed.

Mediumeurlex bulkA stray sector-1/6 work in a legislation dump claims full-sector watermark coverage

eurlex/bulk.py:277–305 — marks is keyed by CELEX sector digit with no sector-3 restriction, so one treaty/judgment in a LEG dump writes the treaties/caselaw watermark from a one-document baseline; a later incremental then skips that sector's entire un-harvested history. Sibling MEDIUM: dump-supplied CELEX strings reach filesystem path construction with no shape validation (bulk.py:255–259 — a / or .. escapes root; a literal _ breaks slug reversibility), while the live download path regex-validates every CELEX.

Fix: write watermarks only for the sector the dump covers; validate CELEX against the sector regexes and count rejects under skipped.

Theme 2 — The render freshness model can't see cross-document edits

Highlib · buildKommentar, guidance, remisser and correspondence layers never reach the pages that render them

lib/render.py:85–193 · build.py:2026–2160 · lib/catalog.py:943–994 — the per-page freshness key covers a page's own artifact + its own sidecars; the dependency digest is identity-based (link sets). But kommentar prose, kommentar .ann guidance, remisser .ann analyses and .corr correspondence render onto other documents' pages. Verified end-to-end: editing kommentar prose never re-renders the host page; ai-annotate opens the generate gate but every page is then judged fresh (0 rendered — the in-code comment claiming a full generate propagates it is wrong for the unforced case); remisser artifacts appear in neither the watermark nor any signature. Only --force or an unrelated change ships these edits.

Fix: per-host-uri content digests for Site's cross-document indexes, folded into page signatures; add remisser globs to generate_watermark. rule:artifact-is-truth

Highbuild · apiThe inline editor's post-commit rebuild doesn't actually make the edit live

lib/render.py:1964–1967 vs build.py:1677–1680 — generate_site ignores only whenever source is also passed, and rebuild_after_commit passes both. Every editor checkout therefore runs a whole-source freshness scan synchronously inside the web request — and, combined with the finding above, then judges the host page fresh, so the committed edit does not appear on the live page, contradicting REWRITE §6. The pinned test mocks cmd_generate and only asserts argument shapes, so nothing catches either half.

Fix: make only compose with source; have the editor path bypass fresh() for pages that are dirty by construction. Bonus: routing write_artifact through write_atomic (Theme 3) also fixes the editor path 500-ing on a first-ever commentary (missing parent dir, after the git commit landed).

MediumlibRepeal status evaluated against today's date, but no freshness key contains a date

lib/render.py:1043, lib/facets.py:379 — when a statute's upphavandedatum passes, nothing invalidates: the site keeps presenting a now-repealed statute as in force until an unrelated change or --force. rule:respect-source-temporality

Fix: fold repeal dates falling in (last_run, today] into generate_watermark and the affected pages' signatures.

MediumbuildRecipe-version tuples miss transitive imports — editing shared code doesn't re-stale its stages

build.py:1727–1729, 2012–2015 — RELATE_CODE omits concepts.py/text.py/markdown.py that catalog.py imports; GENERATE_CODE omits facets.py and api/app.py (which produce the browse pages, via the sanctioned inversion); INDEX_CODE omits catalog.py. Change a facet bucket rule → "up to date — skipped" → stale browse pages until --force.

Fix: extend the tuples to the real import closure, or derive them from the import graph in one place.

Theme 3 — Atomicity: the consolidation missed the most important writer

Mediumbuild · libwrite_artifact is plain write_text; a zero-byte artifact silently drops the document from the catalog

build.py:169 — the one writer the 07-01 write_atomic consolidation missed is the central artifact writer. write_text truncates first: an OOM-killed worker or ENOSPC leaves a zero-byte artifact, which catalog.rebuild (lib/catalog.py:538–541) interprets as a deliberate SkipDocument placeholder and silently drops from the catalog. Same non-atomic in-place writes on the served tree (render.py:1875,1927,2038, dump.py:28) — a reader mid-regenerate can receive a truncated page.

Fix: route all through util.write_atomic. Batch with the other new non-atomic sites found this pass: sfs/versions.py:157 (version artifacts — the sidecar 3 lines below is already atomic), dv/identity.py:233 (the identity index that drives the whole dv parse phase), dv/namedcases.py:90, eurlex/annotate.py:107 and remisser/ai_analyze.py:175 (truncated .ann breaks every later generate), plus the two wiki sites already on the 07-02 list. rule:second-use-goes-to-lib

Theme 4 — Robustness against hostile or pathological input

Mediumlib/lagrumQuadratic blowup in the LAGRUM trigger regex — reproduced: 6.2 s on 24 KB

lib/lagrum.py:473–476 — the unbounded list-continuation quantifier makes the trigger scan O(n²)+ in the length of a digit/separator run (measured 0.09→0.45→1.4→6.2 s per input doubling). Any remote-supplied DV/förarbete text containing a long flattened enumeration (one § anywhere defeats FILTER_LAW) stalls the parse for minutes-to-hours — a hang the per-document exception boundary cannot catch.

Fix: bound the repetition (e.g. {0,50}; WINDOW=220 already caps what the parser can consume). Measured 1.1 s over 300 KB with the bound; no golden-visible behavior change.

MediumapiNo rate limiting on login; every attempt costs a full pbkdf2

api/auth.py:174–194 — no throttle, backoff or lockout on password guesses, and every POST — including unknown usernames, deliberately checked against a dummy hash — burns 260k pbkdf2 rounds in Starlette's bounded sync threadpool. A login flood pins CPU and starves the whole (otherwise cheap) site. The only MEDIUM from the security slice; everything else on the auth/edit/git surface checked out sound (timing-safe compares, -- git argument hygiene, _contained path confinement, CSRF posture, prior XSS/XXE fixes present).

Fix: per-IP + per-account rate limiting with backoff; cap concurrent login work. Security LOWs: version param allows one .. segment where basefile forbids it (app.py:387); anonymous /document/diff has no resource cap (cache it — versions are immutable); sessions unrevocable for 14 days; cookie Secure inferred from a proxy header.

MediumavgLibreOffice conversion can import the previous case's decision under the current dnr

avg/legacy.py:146–230 — every conversion targets tmpdir/index.pdf in one shared temp dir; soffice sometimes exits 0 without producing output, in which case the previous case's stale PDF passes the existence assert and is imported as the current case's body — silent wrong data.

Fix: out.unlink(missing_ok=True) before the run, or per-case subdirectories.

Mediumeurlex · wiki · remisserLLM-response validation gaps defer failures past the retry loop

eurlex/annotate.py:56–74 — recitalGroups/articleToRecitals shapes never checked, so a malformed-but-JSON reply is written to the .ann and crashes every subsequent generate (render.py:1355–1368) instead of being fed back to the model. Sibling LOW: wiki/annotate.py:240 uses unvalidated link["page"] in "#page=%d", crashing after the expensive call. Mediumforarbete · avg · remisserThe load-bearing-assert pattern keeps re-accreting in new code

The exact defect class fixed in 07-01 (§3.3) and listed 5× in the 07-02 punch list has new instances in each newer vertical: forarbete/legacy_formats.py:101,252 (under -O, mints "None:None" basefiles written to disk as junk catalog documents), avg/legacy.py:212–215 (mismatched dnr/date flows into the record), remisser/download.py:240–243 (org-slug collision silently overwrites one org's answer PDF with another's — and the assert is test-pinned, so the fix must adjudicate test_remisser.py:200). rule:errors-drive-retry-use-raise

Theme 5 — SFS validation-harness integrity

Highsfsload_inputs crashes on every legacy-HTML call — and validate silently misclassifies those docs

sfs/__init__.py:55 — Path(json_path).exists() raises TypeError when json_path is None, which is what every legacy-HTML caller passes (reproduced). The documented CLI mode crashes; worse, in corpus validate the TypeError is swallowed by the per-doc boundary, so every HTML-fallback document is misclassified as error: TypeError instead of golden-compared — those docs silently drop out of the correctness net. Production build.py always passes real Paths, so only the CLI/harness is affected — but the harness is the regression net. Related MEDIUM: the legacy register path is derived by a fossil /downloaded/→/register/ transform matching no current layout (sfs/__main__.py:112,134, _validate.py:117), so HTML-path parses silently lose all register-derived amendments/metadata; derive via layout.sfs_sfsr.

Mediumsfsversions stage: a transient environment failure becomes a permanent sidecar skip

sfs/versions.py:147 — the sanctioned per-version boundary catches all Exception, so an OSError from a flaky mount is recorded exactly like a corrupt archive; since the inputs are immutable, the stage is watermark-fresh forever and that historical lydelse silently vanishes from the version panel until the recipe hash changes. The noqa rationale covers data corruption, not environment. rule:narrow-what-you-catch

Fix: re-raise OSError/MemoryError; keep the recorded skip for data-shaped failures.
